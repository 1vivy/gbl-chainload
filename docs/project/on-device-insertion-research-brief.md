# Research brief: on-device cache/profile insertion methods

Audience: research agent investigating the next PR after `feature/objectives-implementation`.

## Goal

Find a clever, safe insertion method for two payload classes:

1. **Cached ABL payload**: a patched LinuxLoader/ABL PE that gbl-chainload can start without re-running dynamic patches.
2. **Mode-2 profile payload**: profile data derived from stock vbmeta and related per-OTA security metadata.

The method must support on-device preparation after OTA, but it must also be testable before any persistent non-HLOS write.

## Current constraints

- Do **not** rely on compiling EDK2 on-device.
- Do **not** require agent-run non-HLOS flashing. User-run recovery/app flows are allowed only after explicit validation.
- Current safe agent test path remains RAM load only:

  ```text
  fastboot stage dist/<artifact>.efi
  fastboot oem boot-efi
  ```

- Current static cache-ABL PR path embeds a host-prepared cached ABL container into the EFI at build time.
- ZIP/recovery delivery was intentionally removed from the current PR; do not assume it is the final UI.
- `patch1-efisp-recursion` is safety-critical for cached/second-stage ABLs. Final prepared cached ABL must not contain UTF-16 `efisp` (`65 00 66 00 69 00 73 00 70 00 00 00`) unless a better, proven recursion guard replaces this requirement.
- Runtime currently uses `LoadImage(FALSE, gImageHandle, NULL, Pe, PeSize, &ImageHandle)` for the cached ABL. `LoadImage` is now performed before protocol hooks; hooks are installed immediately before `StartImage`.

## What needs solving

### A. Payload insertion target

Evaluate ways to carry cached ABL and mode-2 profile data without rebuilding EDK2 on-device:

- Appended EFI overlay with footer/header magic.
- PE section injection or adding a new PE section.
- Replacing a fixed-size data blob in `.data` / `.rdata` by magic.
- External sidecar file on EFISP next to `gbl-chainload.efi`.
- Dedicated small FAT file/container on EFISP.
- GPT/partition-backed sidecar where gbl-chainload reads from known safe storage.
- UEFI variable or misc/oplusreserve-backed small profile only, if suitable.
- Hybrid: static EFI + sidecar profile + cached ABL overlay.

For each candidate, answer:

- Can it be generated on-device with normal recovery/app userspace tools?
- Can it be validated byte-for-byte before writing EFISP or ABL?
- Can the exact same artifact be tested by `fastboot stage`/`boot-efi`, or does it require EFISP context?
- Does UEFI `LoadImage` accept the resulting EFI if bytes are appended or sections are changed?
- Does Secure Boot / image authentication care about appended data in this environment?
- How does gbl-chainload discover its own file or sidecar robustly when launched from EFISP vs staged RAM?
- What is the max payload capacity and EFISP size risk?
- What happens on power loss midway through insertion?
- What are the exact rollback artifacts and recovery steps?

### B. Testability before flashing

Prioritize methods that allow a user to verify behavior without persistent writes.

Research test strategies:

- A staged EFI that contains the exact same inserted payload bytes as the EFISP artifact.
- A staged EFI plus a RAM-disk/fastboot-download sidecar, if gbl-chainload can consume staged memory.
- A self-test command/mode inside gbl-chainload that validates the container/profile and exits without starting ABL.
- A dry-run recovery/app command that produces final files and hashes without writing EFISP/ABL.
- A two-phase install:
  1. prepare artifact + manifest + hashes under `/sdcard/gbl-chainload/`
  2. user explicitly applies after reviewing validation output

The ideal answer includes a pre-flash command sequence and expected logs.

### C. On-device preparation without EDK2 compile

Research practical on-device toolchains and algorithms:

- Can existing host tools (`fv-unwrap`, `abl-patcher`) be cross-compiled as static Android/recovery binaries?
- Is `liblzma` available in the target recovery, or should `fv-unwrap` use a bundled/static decompressor?
- Can a small Python script run in recovery, or should everything be C/BusyBox shell?
- What minimal PE/container patching logic is needed if using section/blob replacement?
- Can mode-2 profile preparation be a simple binary/CBOR/TLV payload generated by shell/C without XML parsing?

### D. Unified container format

Prefer a container that can carry both cached ABL and profile data.

Investigate a versioned TLV layout such as:

```text
magic: GBL_CHAINLOAD_PAYLOAD_v1
version
total_size
entry_count
entries:
  type = cached_abl | mode2_profile | manifest | signature
  offset
  size
  sha256
payload bytes
footer magic / length
```

Required properties:

- Atomic-ish write pattern or temp file + rename where possible.
- Strong hash coverage for every payload.
- Explicit mode compatibility (`mode=1`, `mode=2`).
- Source metadata: OTA ABL hash, extracted PE hash, patched PE hash, profile input hashes.
- Human-readable manifest/log emitted alongside binary container.

### E. Runtime reading path

If using appended overlay or sidecar, determine the most reliable UEFI runtime access path:

- `EFI_LOADED_IMAGE_PROTOCOL` → `DeviceHandle` → `EFI_SIMPLE_FILE_SYSTEM_PROTOCOL` → reopen own file.
- Use `LoadedImage->FilePath` to recover the exact filename instead of hardcoding `\gbl-chainload.efi`.
- Fallback sidecar search order and failure behavior.
- Behavior when launched via `fastboot stage` where there may be no filesystem-backed loaded image.

Need evidence from EDK2/Qcom behavior and/or small test EFI logs.

## Deliverables requested from research agent

Return a ranked recommendation with:

1. Best insertion method for cached ABL.
2. Best insertion method for mode-2 profile.
3. Whether one unified container should serve both.
4. Exact pre-flash test plan.
5. Required on-device helper binaries/scripts.
6. Failure/rollback model.
7. Security/recursion gates.
8. Minimal implementation plan for a separate PR.

Use confidence levels and call out unknowns that need Ghidra/device validation.

## Known evidence to reuse

- `docs/project/re-findings.md` documents LinuxLoader's EFISP GBL probe and recursion hazard.
- Static cache-ABL generation currently rejects final patched ABL payloads that still contain UTF-16 `efisp`.
- A previous staged cache-ABL smoke test worked with `fastboot stage` + `boot-efi`, but that does **not** validate EFISP self-file reopening.
- Ghidra project annotations exist for `LinuxLoader_infiniti.efi` around `LinuxLoaderEntry`, `FindPartitionHandlesByLabel`, and `ReadWholePartitionByLabel`.
